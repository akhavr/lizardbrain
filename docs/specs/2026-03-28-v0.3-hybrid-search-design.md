# lizardbrain v0.3.0 — Hybrid Search & Vector Embedding

**Date:** 2026-03-28
**Status:** Approved (pending spec review)
**Version:** 0.3.0 (no breaking changes)

## Goal

Bring the proven Levi memory architecture (hybrid FTS5 + vector kNN search, embedding pipeline) into the universal lizardbrain package. Model-agnostic, adapter-based, works for any group chat and any LLM/embedding provider.

## Context

lizardbrain v0.2.0 extracts structured knowledge (members, facts, topics) from group chat messages via any OpenAI-compatible LLM and stores it in SQLite with FTS5 full-text search. It works, but search quality is limited to keyword matching.

Levi's production deployment on LIMB.AI/TECH has been running a more advanced architecture for months: vector embeddings via sqlite-vec, hybrid search merging FTS5 + vector kNN via Reciprocal Rank Fusion (RRF). This design generalizes that architecture into lizardbrain.

### What exists today (v0.2.0)

- Modular: adapters (sqlite, jsonl), enrichers (url), config-driven
- Zero dependencies — shells out to system `sqlite3` CLI
- FTS5-only search
- OpenAI-compatible LLM abstraction
- CLI: `init`, `extract`, `stats`, `search`, `who`, `roster`

### What's missing

- Vector tables (sqlite-vec, vec0)
- Embedding pipeline (any OpenAI-compatible `/v1/embeddings` endpoint)
- Hybrid search (FTS5 + kNN + RRF merge)
- Reliable in-process DB access (`better-sqlite3`)
- Metadata tracking for embedding model/dimensions

## Architecture

```
[Chat Source] → [Adapter] → [LLM Extraction] → [SQLite + FTS5]
                                                       |
                                          (if better-sqlite3 + sqlite-vec)
                                                       ↓
                                               [Embedding Pipeline] → [vec0 tables]
                                                       ↓
                                               [Hybrid Search: FTS5 + kNN + RRF]
```

### Two tiers, one codebase

| | Core tier | Vector tier |
|---|---|---|
| **Deps** | System `sqlite3` CLI | `better-sqlite3` + `sqlite-vec` |
| **Install** | `npx lizardbrain init` | `npm install lizardbrain` |
| **Search** | FTS5 (keyword/BM25) | Hybrid: FTS5 + kNN + RRF merge |
| **Extraction** | Same | Same |
| **Embeddings** | None | Auto-backfill after extraction |

Detection is automatic at startup: try `require('better-sqlite3')`, try loading `sqlite-vec`. Set capability flags. Everything downstream checks them.

## Design Details

### 1. Driver Abstraction (`driver.js`)

Replaces `db.js`. Tries `require('better-sqlite3')` at startup, falls back to `sqlite3` CLI.

**Interface:**

```js
const driver = createDriver(dbPath, options);

driver.read(sql, params)        // → array of objects
driver.write(sql, params)       // → { changes: N }
driver.run(sql, params)         // → run without return
driver.transaction(fn)          // → atomic batch (noop wrapper for CLI)
driver.close()                  // → close connection

// Capability flags (read-only)
driver.capabilities.inProcess     // true if better-sqlite3
driver.capabilities.vectors       // true if sqlite-vec loaded
driver.capabilities.transactions  // true if better-sqlite3
```

**BetterSqliteDriver:**
- Opens DB once, holds connection for process lifetime
- Loads `sqlite-vec` extension if available (sets `capabilities.vectors = true`)
- Uses parameterized queries (no SQL escaping needed)
- WAL mode enabled
- Supports real transactions

**CliDriver:**
- Shells out to `sqlite3` per query (current `db.js` behavior, cleaned up)
- No vectors, no real transactions
- `driver.transaction(fn)` just calls `fn()` sequentially
- `esc()` helper for string escaping

**Important:** Only one driver type is active per process. Never both touching the same DB simultaneously. The driver is created once at startup and passed to all modules.

**Migration from `db.js`:** Every call site currently doing `db.read(dbPath, sql)` becomes `driver.read(sql, params)`. The driver holds the DB path internally after creation.

### 2. Embedding Pipeline (`embeddings.js`)

OpenAI-compatible embedding client. Works with any provider that supports `POST /v1/embeddings`.

**Config (in `lizardbrain.json`):**

```json
{
  "embedding": {
    "enabled": true,
    "baseUrl": "https://api.openai.com/v1",
    "apiKey": "sk-...",
    "model": "text-embedding-3-small",
    "dimensions": null
  }
}
```

- `enabled` — must be explicitly `true` to activate vector tier. No silent defaults.
- `baseUrl` — required. NOT defaulted from `llm.baseUrl` (many providers support chat but not embeddings).
- `apiKey` — required. Can be same or different from extraction LLM key. Also reads `LIZARDBRAIN_EMBEDDING_API_KEY` env var.
- `model` — required. Provider-specific model name.
- `dimensions` — optional. Auto-detected from first API response if omitted. Stored in `lizardbrain_meta`.

**Batching:** Token-aware. Estimates ~4 chars/token, targets ~8000 tokens per batch (conservative for most providers). Configurable via `embedding.batchTokenLimit`.

**Rate limiting:** Retry with exponential backoff on 429. Max 3 retries.

**What gets embedded:**

| Source | Text | vec0 table |
|---|---|---|
| facts | `content` field | `facts_vec` |
| topics | `name + ": " + summary` | `topics_vec` |
| members | `display_name + " — " + expertise + " | " + projects` | `members_vec` |

**Backfill flow:**
1. Query records without corresponding vec0 entry (LEFT JOIN on id)
2. Batch embed via API
3. Insert into vec0 tables in a transaction
4. On failure: transaction rolls back, no partial state. Next run retries.

**Idempotency:** Each vec0 row uses the source table's primary key. Inserting a duplicate key is a no-op (OR IGNORE). Safe to retry.

**CLI:**

```
lizardbrain embed --backfill              # embed all unembedded records
lizardbrain embed --stats                 # show embedding coverage
lizardbrain embed --rebuild               # drop vec0 tables, re-embed everything
```

### 3. Metadata Table (`lizardbrain_meta`)

Tracks embedding configuration to detect model changes.

```sql
CREATE TABLE IF NOT EXISTS lizardbrain_meta (
  key TEXT PRIMARY KEY,
  value TEXT,
  updated_at TEXT DEFAULT (datetime('now'))
);
```

Stores:
- `embedding_model` — model name from config
- `embedding_dimensions` — detected or configured dimensions
- `embedding_base_url` — provider URL
- `schema_version` — for future migrations

**Model change detection:** On `embed --backfill`, compare config model against stored `embedding_model`. If they differ, warn and require `--rebuild` flag to proceed. Prevents silently mixing vectors from different models.

### 4. Schema Changes (`schema.js`)

**Core tables:** Unchanged (members, facts, topics, extraction_state, FTS5 indexes + triggers).

**Vec0 tables (created by `embed --backfill` when vectors enabled):**

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS facts_vec USING vec0(
    fact_id INTEGER PRIMARY KEY,
    embedding float[{dimensions}]
);

CREATE VIRTUAL TABLE IF NOT EXISTS topics_vec USING vec0(
    topic_id INTEGER PRIMARY KEY,
    embedding float[{dimensions}]
);

CREATE VIRTUAL TABLE IF NOT EXISTS members_vec USING vec0(
    member_id INTEGER PRIMARY KEY,
    embedding float[{dimensions}]
);
```

**Creation timing:** Vec0 tables are created on first `embed --backfill` when dimensions are known (from config or auto-detect). Not during `lizardbrain init` — core init stays zero-dep.

**Failure handling:** Table creation + first batch insert happen in a single transaction. If embedding API fails, nothing is written. No partial schema states.

**Migration path:** Existing v0.2.0 `lizardbrain.db` files work as-is. Vec0 tables are purely additive.

### 5. Hybrid Search (`search.js`)

**Algorithm:**

1. Parse query text
2. Run FTS5 MATCH against `facts_fts`, `topics_fts`, `members_fts` (BM25 ranking)
3. If vectors available: get query embedding via API, run kNN against `facts_vec`, `topics_vec`, `members_vec`
4. Merge via RRF (Reciprocal Rank Fusion, K=60):
   ```
   score(doc) = SUM( 1 / (K + rank_in_result_set) )
   ```
5. Sort by merged score, return top N

**Fallback:** If no vectors → pure FTS5 search. Same result format.

**Why RRF:** Deterministic, no tuning parameters, handles the mismatch between sparse BM25 scores and dense cosine similarity. K=60 is standard from the original RRF paper. Proven in Levi's production for months.

**Result format:**

```js
{
  mode: "hybrid",           // or "fts5"
  results: [
    {
      source: "fact",       // fact | topic | member
      id: 42,
      text: "Claude Opus costs $15/M output tokens",
      score: 0.0312,        // RRF score (hybrid) or BM25 rank (fts5)
      confidence: 0.95,     // only for facts
      member: "Alice",      // source member if applicable
      tags: "pricing, claude"
    }
  ]
}
```

**CLI:**

```
lizardbrain search "RAG pipeline"                # auto-picks best available method
lizardbrain search "RAG pipeline" --limit 10     # top 10
lizardbrain search "RAG pipeline" --json         # JSON output (for agent plugins)
lizardbrain search "RAG pipeline" --fts-only     # force FTS5 even if vectors available
```

**Programmatic:**

```js
// Using the driver (recommended — driver already knows the DB path)
const driver = lizardbrain.createDriver('./memory.db');
const { mode, results } = lizardbrain.search(driver, 'RAG pipeline', { limit: 5 });

// One-shot convenience (opens driver internally, closes after)
const { mode, results } = lizardbrain.search('./memory.db', 'RAG pipeline', { limit: 5 });
```

### 6. Pipeline Changes

**`extract` auto-embeds:**

Current: `extract` writes facts/topics/members → done.
New: `extract` writes facts/topics/members → if embedding configured, runs backfill for new records.

- Embedding is best-effort. If it fails, extraction results are still committed. Embedding failure does NOT rollback extraction.
- `--no-embed` flag skips the post-extraction embed step.
- Single cron line does everything:

```bash
0 */2 * * * cd /project && LIZARDBRAIN_LLM_API_KEY=key LIZARDBRAIN_EMBEDDING_API_KEY=key lizardbrain extract >> /tmp/lizardbrain.log 2>&1
```

## What's NOT Changing

- `llm.js` — extraction prompt, OpenAI-compatible client, response parsing
- `adapters/sqlite.js`, `adapters/jsonl.js` — source adapters
- `enrichers/url.js` — URL metadata enrichment
- Core schema tables — members, facts, topics, extraction_state
- Dedup logic in `store.js` — exact prefix + FTS keyword overlap
- Config format for existing fields

## Deferred to v0.4.0

- **Raw message embedding** (`messages_vec`) — needs idempotency, cleanup, dedup/weighting rules vs extracted facts
- **Message lifecycle** — edits, deletions, compaction, stale vector cleanup
- **Reactions table** — unproven in production
- **Historical import** — too platform-specific (Telegram JSON) for core

## Risk Assessment

| Risk | Mitigation |
|---|---|
| `better-sqlite3` native compilation fails on some platforms | Ships prebuilds for major platforms. Fallback to CLI tier. |
| `sqlite-vec` not available for target platform | Graceful degradation: `capabilities.vectors = false`, FTS5 search works fine |
| Embedding API outage during extract+embed | Embedding is best-effort. Extraction commits independently. Next cron retries. |
| User changes embedding model mid-deployment | Model change detection warns. `--rebuild` required. No silent mixing. |
| Search quality variance between tiers | `mode` field in results. `--fts-only` flag. Stats show current tier clearly. |
